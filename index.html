浏览器的回流与重绘

Posted on 2019-07-21 | Edited on 2019-07-24

回流与重绘(Reflow & Repaint)

在讨论回流与重绘之前, 我们要先知道如下几点:

  1. 浏览器使用流式布局模型(Flow Based Layout)
  2. 浏览器会把 HTML 解析成 DOM , 把 CSS 解析成 CSSOM , DOM 和 CSSOM 合并就产生了 RenderTree
  3. RenderTree 中包含了所有节点的样式, 然后计算他们在页面上的大小和位置, 最后把节点绘制到页面上
  4. 由于浏览器使用流式布局, 对 RenderTree 的计算通常只需要遍历一次就可以完成, 但 table 以及内部元素除外, 他们可能需要多次计算, 也就是说需要等同于其他元素要多花几倍的时间, 这也就是为什么要避免使用 table 布局的原因之一

一句话: 回流必将引起重绘, 重绘不一定会引起回流

回流 (Reflow)

当 RenderTree 中部分或者全部元素的尺寸/结构/属性发生改变时, 浏览器重新渲染部分或者全部文档的过程叫做回流

导致回流的操作:
  • 页面首次渲染
  • 浏览器窗口大小发生改变
  • 元素尺寸或者位置发生改变
  • 元素内容的变化 (文字数量或者图片大小等等)
  • 元素字体大小变化
  • 添加或者删除可见的 DOM 元素
  • 激活 CSS 伪类 (例如: :hover)
  • 查询某些属性或者调用某些方法
一些常用并且会导致回流的属性和方法:
  • clientWidth、clientHeight、clientTop、clientLeft
  • offsetWidth、offsetHeight、offsetTop、offsetLeft
  • scrollWidth、scrollHeight、scrollTop、scrollLeft
  • scrollIntoView()、scrollIntoViewIfNeeded()
  • getComputedStyle()
  • getBoundingClientRect()
  • scrollTo()

重绘 (Repaint)

当页面中元素样式的改变并不影响它在文档流中的位置时（例如：color、background-color、visibility等），浏览器会将新样式赋予给元素并重新绘制它，这个过程称为重绘。

性能影响

回流比重绘的代价要更高。

有时即使仅仅回流一个单一的元素，它的父元素以及任何跟随它的元素也会产生回流。
现代浏览器会对频繁的回流或重绘操作进行优化：

浏览器会维护一个队列，把所有引起回流和重绘的操作放入队列中，如果队列中的任务数量或者时间间隔达到一个阈值的，浏览器就会将队列清空，进行一次批处理，这样可以把多次回流和重绘变成一次。

当你访问以下属性或方法时，浏览器会立刻清空队列：

  • clientWidth、clientHeight、clientTop、clientLeft

  • offsetWidth、offsetHeight、offsetTop、offsetLeft

  • scrollWidth、scrollHeight、scrollTop、scrollLeft

  • width、height

  • getComputedStyle()

  • getBoundingClientRect()

因为队列中可能会有影响到这些属性或方法返回值的操作，即使你希望获取的信息与队列中操作引发的改变无关，浏览器也会强行清空队列，确保你拿到的值是最精确的。

如何避免

CSS
  • 避免使用 table 布局。
  • 尽可能在 DOM 树的最末端改变 class。
  • 避免设置多层内联样式。
  • 将动画效果应用到 position 属性为 absolute 或 fixed 的元素上。
  • 避免使用 CSS 表达式（例如：calc()）。
JavaScript
  • 避免频繁操作样式，最好一次性重写 style 属性，或者将样式列表定义为 class 并一次性更改 class 属性。
  • 避免频繁操作 DOM，创建一个 documentFragment，在它上面应用所有 DOM 操作，最后再把它添加到文档中。
  • 也可以先为元素设置 display: none，操作结束后再把它显示出来。因为在 display 属性为 none 的元素上进行的 DOM 操作不会引发回流和重绘。
  • 避免频繁读取会引发回流/重绘的属性，如果确实需要多次使用，就用一个变量缓存起来。
  • 对具有复杂动画的元素使用绝对定位，使它脱离文档流，否则会引起父元素及后续元素频繁回流。

什么是 cookie

cookie 是存储于访问者计算机中的变量. 每当同一台计算机通过浏览器请求某个页面时, 就会发送 cookie , 你可以使用JavaScript来创建和取回 cookie 的值

在JavaScript中通过 document.cookie 来读取或者设置 cookie , 由于 cookie 多用在客户端和服务端之间进行通信, 所以除了JavaScript以外, 服务端的语言 (例如PHP) 也可以存取 cookie

cookie介绍

  1. cookie 是有大小限制的，每个 cookie 所存放的数据不能超过4kb，如果 cookie 字符串的长度超过4kb，则该属性将返回空字符串。
  2. 由于 cookie 最终都是以文件形式存放在客户端计算机中，查看和修改 cookie 都是很方便，所以 cookie 不能存放重要信息。
  3. 每个 cookie 的格式都是这样的：<名>=<值>；名称和值都必须是合法的标示符。
  4. cookie 是存在有效期的。在默认情况下，cookie 的生命周期是会话级别的。如果想要 cookie 能在浏览器关掉之后还可以使用，就必须要为该 cookie 设置有效期，也就是 cookie 的失效日期。
  5. cookie 有域和路径的概念。域就是domain，因为浏览器是个注意安全的环境，所以不同的域之间是不能互相访问 cookie 的(可以通过特殊设置的达到 cookie 跨域访问)。路径就是routing的概念，一个网页所创建的 cookie 只能被与这个网页在同一目录或子目录下得所有网页访问，而不能被其他目录下得网页访问。

cookie常见的问题

  1. cookie 存在两种类型：

    • 当前浏览网站本身设置的 cookie
    • 第三方网页嵌入的 cookie

  2. cookie 的两种清除方式：

    • 通过浏览器工具清除 cookie

    • 通过设置 cookie 的有效期来清除

      注意：删除 cookie 可能会导致某些网页无法正常运行

  3. 浏览器可以通过设置来接受或者拒绝访问 cookie

webpack优化配置

production模式打包自带的优化

  • tree shaking

tree shaking 是一个术语, 通常用于打包时移除 JavaScript 中未引用的代码 (dead-code), 它依赖于 ES6模块系统中的 import 和 export 静态结构特性。

开发时引入一个模块后, 如果只使用其中一个功能, 上线打包时只会把用到的功能打包进文件, 其他没用到的功能不会打包进来, 可以实现最基础的优化

  • scope hoisting

    Scope hoisting的作用是将模块之间的关系进行结果推测, 可以让 webpack打包出来的代码文件更小, 运行更快

    scope hoisting 的实现原理就是在不造成代码冗余的前提下分析模块之间的依赖关系把零碎的模块合并到一个函数中, 由于 scope hoisting需要分析出模块之间的依赖关系 因此源码必须采用 ES6模块化语法

  • code compression

    所有代码使用UglifyJsPlugin插件进行压缩 混淆

  • 将css提取到独立文件中

js优化

Code splitting 是 webpack打包时用到的重要的优化特性之一, 此特性能够把代码分离到不同的 bundle文件中, 然后可以按需加载或者并行加载这些文件。代码分离可以用于获取更小的 bundle, 以及控制资源加载优先级, 如果使用合理, 会极大影响加载时间。

有三种常用的代码分离方法 :

  • 入口起点 (entry points) : 使用 entry 配置手动地分离代码
  • 防止重复 (prevent duplication) : 使用 SplitChunksPlugin 去重 然后分离 chunk
  • 动态导入 (dynamic imports) : 通过模块的内联函数调用来分离代码

  1. 在 webpack.config.js 配置文件中设置多个入口

  2. 这种方法打包完会存在一些问题 :

    • 这种方法不灵活, 并且不能讲核心应用程序逻辑进行动态拆分代码

提高构建性能

noParse

在引用一些第三方模块时, 例如 jQuery/bootstrap等, 我们知道这两个内部不会有相互依赖, 因为最终我们用到的只是一个单独文件, 而 webpack会去解析他们内部依赖关系, 我们需要阻止webpack的默认行为.

可以在 webpack配置文件中的 module 节点下加上 noParse , 并配置正则来确定不需要判断的依赖关系的模块

lgnorePlugin

在引入一些第三方模块时, 例如 moment, 内部会做国际化处理, 所以会包含很多语言包, 而语言包打包时会比较占空间, 如果我们项目只需要少数语言, 可以忽略掉其他的所有语言包, 然后按需引入语言包从而使得构建效率更快, 打包生成的体积更小

需要忽略第三方模块内部依赖的其他模块, 只需要做如下处理 :

  1. 安装并找到 moment依赖的语言包

  2. 使用 ignorePlugin 插件忽略其依赖

  3. 按需手动引入

DellPlugin

在引用一些第三方模块时, 例如vue/react/angular等框架, 这些框架的文件一般都是不会修改的, 而每次打包都需要去解析它们, 也影响打包速度, 哪怕做拆分, 也只是提高了上线后用户访问速度, 并不会提高构建速度, 所以如果需要提高构建速度, 应该使用动态链接库的方式, 类似于 Windows的 dll 文件

借助 DllPlugin插件实现将这些框架作为一个个的动态链接库, 只构建一次, 以后每次构建都只生成自己的业务逻辑代码, 可以大大提高构建效率!

其主要的原理在于, 将一些不做修改的依赖文件, 提前打包, 这样我们开发代码发布的时候就不需要再对这部分代码重新进行打包, 从而节省了打包时间

涉及两个插件:

  1. DllPlugin

    使用一个单独webpack配置创建一个dll文件。并且它还创建一个 manifest.json 文件, DllReferencePlugin使用该 json文件来做映射依赖性 (这个文件会告诉我们哪些文件是已经提前打包好的)

    配置参数 :

    • context (可选) : manifest文件中请求的上下文, 默认为该webpack文件上下文
    • name : 公开的 dll函数的名称, 和 output.library保持一致即可
    • path : manifest.json 生成的文件夹及名字

  2. DllReferencePlugin

    这个插件用于 webpack配置, 它引用的 dll需要预先构建的依赖关系

    • Context : manifest文件中请求的上下文
    • manifest : DllPlugin插件生成的 manifest.json
    • content (可选) : 请求的映射模块id (默认为 manifest.content)
    • name (可选) : dll暴露的名字
    • scope (可选) : 前缀用于访问dll内容
    • sourceType (可选) : dll是如何暴露 (libraryTarget)

浏览器缓存

在做了众多代码分离的优化后, 其目的是为了利用浏览器缓存, 达到提高访问速度的效果, 所以构建项目时做代码分隔是必要的, 例如将固定的第三方模块抽离, 下次修改了业务代码, 重新发布上线不重启服务器, 用户再次访问服务器就不需要再次加载第三方模块

而这也给我们带来了一个新的问题, 如果再次打包上线不重启服务器, 客户端会将以前的业务代码和第三方模块同时缓存, 再次访问时依旧访问缓存中的业务代码, 所以会导致业务代码无法及时更新

此时需要在 output节点的 filename中使用 placeholder语法, 根据代码内容生成文件名的hash :

之后每次打包业务代码时, 如果有改变, 会生成新的 hash作为文件名, 浏览器就不会使用缓存, 而第三方模块不会重新打包生成新的名字, 则会继续使用缓存, 从而解决了这个问题

Prefetching和Preloading

在优化访问性能时, 除了充分利用浏览器缓存之外, 还需要涉及一个性能指标 : coverage rate (覆盖率)

可以在 chrome浏览器的控制台中按 : Ctrl + shift + p 查找 show coverage 打开覆盖率面板

点击开始录制后刷新网页, 即可看到每个 .js 文件的覆盖率, 以及总的覆盖率

  • 想提高覆盖率, 我们需要尽可能多的使用动态导入, 也就是懒加载功能, 将一切能使用懒加载的地方都使用懒加载, 这样可以大幅提高覆盖率

  • 但有时候使用懒加载会影响用户体验, 所以可以在动态导入时使用魔法注释 Prefetching , 是指在首页资源加载完毕后, 将动态导入的资源默默加载进来, 这样既可以提高首屏加载速度, 也可以解决懒加载可能带来的用户体验差的问题

source map

在webpack配置中发现有时候我们需要查找报错代码在项目中的具体位置, 而webpack打包好的 .js 文件在引入时已经被处理过 此时我们可以通过设置 devtool来转换成 原始代码 (方便开发)

  • 开发环境推荐 cheap-module-eval-source-map

  • 生产环境推荐 none (不使用source map)

  • 原因如下:

    1. 使用 cheap模式可以大幅提高 source map 生成的效率。大部分情况我们调试并不关心列信息, 而且就算 source map没有列, 有些浏览器引擎 (例如 v8) 也会给出列信息。
    2. 使用 module 可支持 babel 这种预编译工具映射转换前的代码
    3. 使用 eval 方式可大幅提高持续构建效率。官方文档提供的速度对比表格可以看到 eval 模式的重新构建速度是很快的 - 详细请看 webpack官网
    4. 使用 eval-source-map 模式可以减少网络请求。这种模式开启 DataUrl 本身包含完整的 source map信息, 并不需要像 sourceURL 那样, 浏览器需要发送完整请求去获取 source map 文件, 这会略微提高效率。而生产环境中则不宜用 eval, 这样会让文件变得极大。

  • 设置 webpack.config.js

      • 可在你确认不需要更新组件时使用

      • 在初始化时不会被调用

      • 在组件完成更新后立即调用
      • 在初始化时不会被调用
    • componentWillUnmount
      • 在组件从DOM中移除的时候立即调用

Promise 介绍

什么是 Promise

  • Promise 是异步编程的一种解决方案，专门处理回调地狱问题
  • Promise 是一个构造函数 其中有 resolve(成功回调) 和 reject(失败回调)
  • Promise 构造函数的 Prototype 属性中有 .then() 和 .catch() - - - [所有实例都能访问]

  • 实现链式编程

    • 由于 Promise 实例是异步操作 调用时会有两种状态
    • 状态 1 : 异步执行成功 内部调用成功回调函数 resolve() 返回结果
    • 状态 2 : 异步执行失败 内部调用失败回调函数 reject() 返回结果

    • 在创建 Promise 实例时调用 .then() 预先为异步操作指定 resolve(成功回调) 和 reject(失败回调)

      注意: [ .then() 必须有成功回调 可以没有失败回调 ]
